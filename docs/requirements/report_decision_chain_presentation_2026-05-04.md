# 报告决策链呈现需求

状态：IN_PROGRESS

最后更新：2026-05-04

关联任务：`REPORT-003`、`REPORT-004`、`DOC-001`、`EXEC-001`、`CHAIN-001`、`UI-001`

## 背景

本轮评估来自 2026-05-04 关于最终报告表现形式的讨论。初步结论是：最终报告不应写成模块结果堆叠，而应写成一条可阅读、可执行、可审计的决策链。

报告读者首先需要理解：

```text
现在处于什么状态？
建议怎么处理仓位？
为什么是这个结论？
哪些证据支持？
哪些因素限制了仓位？
什么情况会改变判断？
这个判断有多可靠？
```

现有系统已经完成 `REPORT-001` 的 evidence bundle / trace lookup，完成 `REPORT-002` 的变化原因树和“什么情况会改变判断”，也已实现评分、置信度和仓位闸门拆分。因此下一阶段重点不是增加更多原始信息，而是把已有信息组织成人能顺着读下去的逻辑链。

## 初步结论

值得登记为开发计划的内容应收敛为两类：

1. 日报和最终报告的决策链呈现规范：结论卡、判断句、章节顺序、状态标签、动作语言和证据下钻边界。
2. 周报/月报复盘模板：日报偏行动，周报偏判断变化，月报偏系统校准和规则学习。

不建议把本轮讨论拆成大量独立任务。评分、置信度、仓位三层拆分、position gate、变化原因树、trace lookup、thesis 状态机等能力已经有对应任务或实现，应作为新呈现模板的输入，而不是重复登记。

## REPORT-003

标题：最终报告决策链呈现规范

价值判断：值得作为 P1 任务。当前日报已经有顶部摘要、变化原因树、认知状态、仓位闸门、模块评分、硬数据信号、人工复核摘要和可追溯引用，但整体仍接近“摘要 + 模块明细表”。下一步应固定成“结论先行、原因展开、约束解释、行动落点、证据下钻”的人类阅读顺序。

目标结构：

```text
1. 今日结论卡
2. 本期变化原因树
3. 产业链认知状态
4. thesis 状态
5. 风险与仓位闸门
6. 评分明细
7. 什么会改变判断
8. 证据与审计下钻
```

结论卡必须回答：

- 状态标签。
- 市场吸引力。
- 判断置信度。
- 评分映射仓位。
- 风险闸门后最终仓位。
- 本期执行动作。
- 一句话主结论。
- 三个核心原因。
- 一个最大限制。
- 一个下一步触发条件。

呈现原则：

- 先给判断句，再给数字、证据和限制条件。
- 评分明细不能放在报告最前；评分是解释输入，不是读者的第一入口。
- 仓位必须拆成评分映射仓位、风险闸门后仓位和执行建议。
- 报告必须明确区分“机会好不好”“判断靠不靠谱”“当前风险约束下能承担多少暴露”。
- 报告应强调动作，但避免写成自动交易指令。
- 正文只放核心证据摘要、claim id、质量状态和复核状态；详细证据通过 evidence bundle / trace lookup 下钻。

建议状态标签：

|状态|含义|典型动作|
|---|---|---|
|积极进攻|趋势、基本面、置信度和风险环境都支持|可加仓|
|中高配|机会仍好，但存在部分约束|持有或小幅加仓|
|中高配但受限|分数不错，但 gate 限制仓位|持有，不追高|
|中性观察|信号分歧或置信度不足|维持，等待确认|
|防守降仓|趋势、风险或 thesis 明显恶化|减仓|
|人工复核|数据不足、风险事件不确定或 thesis 被挑战|暂停自动结论|

验收标准：

- `score-daily` 日报开头生成结论卡，且结论卡位于数据门禁、模块评分和硬数据信号之前。
- 日报用判断句解释主结论，并限制摘要为“一个主结论、三个核心原因、一个最大限制、一个下一步触发条件”。
- 日报同时输出市场吸引力、判断置信度、评分映射仓位、风险闸门后最终仓位和执行动作，不允许把总分直接写成仓位或可靠性。
- 仓位解释能说明为什么分数不低但上限被估值、thesis、置信度、风险事件或组合约束压低。
- 评分明细只解释最重要变化，不把所有指标平铺到正文前部。
- 证据下钻章节只列核心 claim/evidence/dataset/quality refs 和 trace lookup 入口，不重复塞入所有原始数据。
- 实现时同步更新 `docs/system_flow.md`，并补充日报快照或文本顺序测试。

开放问题：

- 状态标签应完全由配置阈值生成，还是允许 `belief_state` 输出推荐标签。
- “三个核心原因”的排序应按仓位影响、分数变化、证据可信度还是固定模块优先级。
- 结论卡是否同时覆盖日报和回测报告，还是先从日报落地。
- 旧的 `render_recommendation_markdown` 是否应并入统一日报模板，避免两套报告语义漂移。

当前实现状态：

- 2026-05-04：进入基础实现；第一阶段先在 `score-daily` 日报开头生成“今日结论卡”，固定输出状态标签、市场吸引力、判断置信度、评分映射仓位、风险闸门后最终仓位、执行动作、主结论、三个核心原因、最大限制和下一步触发条件。
- 第一阶段不新增评分信息，也不改变 scoring、`position_gate`、execution policy 或 evidence bundle 语义，只重排已有日报解释入口。
- 完整 `DONE` 仍需要统一所有报告模板、沉淀可配置状态标签和更精细的核心原因排序策略；当前目标完成态预计为 `BASELINE_DONE`。
- 2026-05-04 基础版已完成：`score-daily` 日报开头新增“今日结论卡”，执行动作从 `execution_policy` advisory 输出进入结论卡；文本顺序测试覆盖结论卡位于变化原因树、数据门禁和模块评分之前。
- 当前完成态为 `BASELINE_DONE`：日报决策链入口已具备；完整 `DONE` 仍需要统一旧的 `render_recommendation_markdown`、周/月报模板、状态标签配置化和核心原因排序策略。

## REPORT-004

标题：周报/月报投资复盘模板

价值判断：值得登记为 P2。日报偏行动解释，现有 `feedback loop-review` 偏系统闭环运营。周报/月报应面向人类投资复核者，分别回答“本周判断是否变化”和“本月系统是否学到东西”。

周报目标结构：

```text
1. 本周结论是否变化
2. 本周仓位是否变化
3. 本周改变判断的前三个证据
4. 产业链节点热度/健康度变化
5. thesis 状态变化
6. 风险事件变化
7. 系统自评：本周判断是否被市场验证
8. 下周最重要的观察事件
```

月报目标结构：

```text
1. 本月系统判断表现
2. 高分信号是否有效
3. gate 是否过严或过松
4. thesis warning 是否有提前量
5. 哪些数据源或模块影响了置信度
6. 候选规则是否需要调整
```

验收标准：

- 周报按周汇总 decision snapshots、变化原因树、thesis/risk/valuation 状态和核心证据变化。
- 月报连接 outcome、calibration、learning queue 和 rule experiment，说明哪些结论已验证、哪些仅研究、哪些因样本不足不能升级。
- 周报/月报声明 `ai_after_chatgpt` 市场阶段和实际请求日期范围。
- 周报/月报继续使用 evidence bundle / trace lookup 作为审计下钻入口，不复制所有原始证据。
- 实现时同步更新 `docs/system_flow.md` 并补充报告文本测试。

开放问题：

- 周报/月报是扩展 `aits feedback loop-review`，还是新增独立 report 子命令。
- 周报是否需要固定自然周，还是允许任意 5 个交易日窗口。
- 月报是否需要等待 `BACKTEST-001` 和真实 replay runner 更完整后再推进。

## 关联任务更新建议

### DOC-001

`DOC-001` 已覆盖 `actionable`、`review_required`、`research_only`、`data_limited`、`backtest_limited` 等使用边界等级。应补充“双层状态”设计：

- 投资姿态标签回答“当前 AI 产业链处于什么投资状态”，例如积极进攻、中高配但受限、中性观察、防守降仓。
- 结论使用等级回答“这个结论能不能作为仓位复核依据”，例如 actionable、review_required、research_only。

两者必须分开，避免把市场状态和结论可靠性混在一起。

### EXEC-001

`EXEC-001` 应吸收动作语言规范。日报动作建议应使用：

- 维持。
- 小幅加仓。
- 禁止主动加仓。
- 减仓到目标区间。
- 等待人工复核。
- 观察，不形成交易结论。

除非项目 owner 明确改变产品边界，否则报告不应默认输出“买入/卖出”式交易指令。

### CHAIN-001

`CHAIN-001` 应补充产业链因果链模板：

```text
市场价格信号
  -> 产业链节点
  -> thesis 状态
  -> 评分模块
  -> 仓位 gate
  -> 行动影响
```

该模板服务报告解释，不应在第一版直接改变仓位。

### UI-001

读者模式更适合放入仪表盘阶段：

- 快速读者：结论卡、动作建议、最大风险、什么会改变判断。
- 投资复核者：变化原因树、产业链节点、thesis 状态、risk gate、仓位上限来源。
- 系统审计者：claim/evidence/dataset/quality refs、trace lookup、数据质量门禁、source policy。

Markdown 日报优先通过章节顺序和附录分层解决；交互式读者模式留给 `UI-001`。

## 暂不登记为新任务的内容

- 评分、置信度、仓位三层拆分：已由 `SCORE-002` 和 `SCORE-001` 覆盖。
- 变化原因树和“什么会改变判断”：已由 `REPORT-002` 覆盖。
- evidence bundle、claim id、trace lookup：已由 `REPORT-001` 覆盖。
- thesis active/warning/challenged/invalidated 状态：已由 `THESIS-001` 覆盖。
- 产业链节点热度/健康度：已有 `CHAIN-001`，只需补充因果链表达。
- 执行纪律：已有 `EXEC-001`，只需补充动作词汇和非自动交易边界。
