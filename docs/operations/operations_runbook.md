# AITradingSystem Operations Runbook

最后更新：2026-06-26

本文是 AITradingSystem 周期性任务的总入口。执行任何 daily、weekly、biweekly、monthly、governance、scheduler validation 或 artifact catalog consistency 任务前，先读取本文，再进入对应的细分 runbook、配置或报告。

本文不替代 `config/scheduled_tasks.yaml`、`config/etf_portfolio/operations_schedule.yaml`、`docs/runbooks/scheduled_task_orchestration.md`、`docs/runbook_daily_ops.md`、`docs/artifact_catalog.md` 或 `docs/system_flow.md`。其中：

- `config/scheduled_tasks.yaml` 是调度 cadence 和命令登记的配置源。
- `config/etf_portfolio/operations_schedule.yaml` 是 TRADING-074 ETF portfolio operations schedule source config，用于描述 ETF research daily / weekly / biweekly / monthly / manual-review command plan、dependencies、expected outputs 和 safety boundary；它不是外部 scheduler entry。
- `docs/runbooks/scheduled_task_orchestration.md` 说明 daily chain、非 daily cadence 隔离和 scheduler safety checklist。
- `docs/runbook_daily_ops.md` 说明人工交接、失败排查和运行归档。
- `docs/artifact_catalog.md` 说明关键 artifact 的生成者、上游、下游、production effect 和常见误解。
- `docs/system_flow.md` 仍是数据输入到结论输出路径的源-of-truth 图。

## Operating Principles

- 每日 scheduler trigger 是统一外部入口。Windows Task Scheduler、cron、GitHub Actions 或云调度器默认只应调用 `aits ops daily-run`，不要把 weekly / biweekly / monthly / governance 命令直接散落成多个未审计系统任务。
- 更长周期任务可以由同一个每日 scheduler 入口根据日期和条件触发，但必须通过受控编排实现：检查交易日、cadence due 状态、上游 daily artifacts、数据质量、production safety 和人工审批条件。
- 当前 baseline 中，`aits ops daily-run` 只执行 `daily_trading_day` 链路；weekly / biweekly / monthly / ad hoc research 任务已在 `config/scheduled_tasks.yaml` 登记，但不会自动进入 daily-run。自动 due-cadence 执行实现前，非 daily 任务由 operator 按本文手动运行或由后续受控 scheduler 执行。
- `scheduled_tasks.yaml` 的 daily task可用 `activation_condition=always|trading_day_only|closed_market_only` 声明 market-session条件；未声明等同 `always`，未知值加载失败。`official_policy_sources` 使用 `closed_market_only`，因此交易日plan不包含它、休市日plan在`validate_data`后包含它；`ops_daily.py`与canonical shadow spec必须使用同一activation解析，不允许重新硬编码另一套条件。
- `aits ops daily-run` 已由 canonical runtime control 包裹既有 daily executor：同一workflow/spec/as-of只能有一个active lease；已完成重复触发不重跑，partial run只恢复明确idempotent且未PASS的步骤，run/step attempt耗尽、并发或不安全恢复均在命令执行前fail closed。运行证据位于`outputs/run_control/daily/states/`，同一idempotency key分别生成`operations_execution_state.v1` state和`run_ledger.v1` ledger；operator不得手工删除state/lock来强行重跑，只有已过期lock可由控制层回收。
- `daily_trading_day` 中允许登记轻量 scheduler gate，只用于 due/skip/block 审计和只读检查。例如 TRADING-099 的 `aits etf dynamic-v3-rescue schedule observe --as-of {as_of}` 只检查 weekly due 条件、latest pointer validation、stale 状态和可选 observe-only shadow monitor，不自动运行 `run-profile`、real sweep、promotion pack、approval、baseline mutation 或 broker action。
- TRADING-074A/B/C/D/E/F/G/H/I/J/K 的 ETF operations schedule spec、daily / weekly / biweekly / monthly graph、artifact freshness checker、failure policy evaluator、owner checklist builder、scheduler dry-run、operations health report、Reader Brief operations health section 和 final validation gate 只用于规划、校验和展示 ETF portfolio research workflow；weekly / biweekly / monthly graph 会显式披露跨 cadence 上游为 external dependencies，并标记人工复核 checkpoint；monthly graph 保证 bounded historical weight search 只在 slower cadence 计划中出现；freshness checker 只读扫描既有 artifacts，required stale/missing/unknown artifacts 标记为 blocking，optional missing artifacts 标记为 warning，并传播 upstream blocking dependency；failure policy evaluator 把 issue 转成 `info` / `warning` / `error` / `critical` severity、`fail_pipeline` / `block_dependent_steps` / `manual_review_required` 等 policy effect 和稳定 summary；owner checklist builder 从 `manual_review_steps` 生成 daily / weekly / monthly / incident owner review template，并绑定 blocking、warning、manual-review events 与 signoff；`aits etf ops dry-run --cadence ... --as-of ...` 只生成 planned steps、blocking failures、warnings、expected outputs 和 safety summary，`aits etf ops report --cadence ... --as-of ...` 只把 graph/freshness/failure/checklist/dry-run 状态写成 JSON / Markdown operations health report；Reader Brief 只读 report index 指向的 latest `etf_operations_health_report_v1`，展示 status、blocking failures、warnings、stale/missing artifacts、next owner review、safety posture 和 detail link，缺失时显示 section-level `MISSING`；`aits etf ops validate --as-of ...` 生成 `etf_operations_validation_v1` JSON / Markdown，校验 A-J 组件可用性、required missing blocker、optional missing warning、Reader Brief registry integration 和固定 safety boundary。这些路径都不执行 planned commands、不写 production state、不自动 dispatch cadence、不改变 `aits ops daily-run` 的统一入口。
- TRADING-367 data refresh audit 使用 `aits validate-data` 写 validation sidecar，随后用 `aits data refresh-audit report --as-of {as_of}` / `validate --latest` 聚合 validation gate、latest market refresh summary、price cache checksum/row count 和 U.S. market calendar skip reason。Paper-shadow 或其他 cached data-dependent 复核在解释 freshness/refresh 证据前应确认 latest audit 可用；缺 validation sidecar 必须 fail closed。该 audit 不刷新数据、不补造 cache、不降低 `aits validate-data` 门禁、不写 official target weights、不触发 broker/order，Reader Brief 只读 latest `data_refresh_audit`。

## Unified Daily Trigger

标准外部调度入口：

```powershell
aits ops daily-run
```

人工预检入口：

```powershell
aits ops daily-plan --fail-on-missing-env
```

指定评估日时使用：

```powershell
aits ops daily-run --as-of YYYY-MM-DD
```

`daily-run` 面向最新已完成的 U.S. equity market trading day。历史严格复现使用 `aits ops replay-day --mode cache-only --as-of YYYY-MM-DD`，不要用生产 daily scheduler 入口补造历史 PIT 视图。

`daily-plan`与`daily-run`会在既有`daily_ops_plan_*.md`旁写入additive `*.operations_shadow.json`。该sidecar只记录activated canonical WorkflowSpec、due resolution、non-executing RunLedger和legacy parity，固定`commands_executed=false`、`non_daily_dispatch_enabled=false`；它不是第二个scheduler入口，也不表示任何step已PASS。Parity不是PASS时必须fail closed。

ARCH-004F1 runtime-control policy位于`config/operations/runtime_control.yaml`。当前sidecar只披露其path/hash、attempt/resume policy和`legacy_daily_executor_cut_in_enabled=false`；因此lock/idempotency state尚未控制生产daily executor。F1.4 cut-in前不得把generic runtime-control测试解释为daily-run已经具备自动resume，也不得手工将policy flag改为true绕过parity与验证。

## Date And Condition Dispatch

未来若把 weekly / biweekly / monthly / governance 任务接入每日 scheduler trigger，必须遵守以下 dispatch 条件：

1. 每日 trigger 仍只暴露一个外部入口，例如 `aits ops daily-run` 或后续明确命名的统一 scheduler entry。
2. due 判断必须基于明确日期规则、交易日历和已完成 daily artifacts，而不是系统启动时间的隐式副作用。
3. 非 daily 任务运行前必须确认当日 daily data gate 和必需上游 artifact 状态；缺失时输出 `SKIPPED`、`LIMITED`、`INSUFFICIENT_DATA` 或 fail-closed 诊断，不补造结论。
4. 任务输出必须写明 cadence、as_of、date range、source artifacts、data quality status 和 `production_effect`。
5. 任一会影响投资解释的阈值、promotion gate、readiness rule 或样本 floor 必须来自配置/政策 manifest，或按 AGENTS 规则登记为可审计 pilot baseline。
